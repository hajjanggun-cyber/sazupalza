# K-Style Hub Unified Writing Command

이 문서는 K-Style Hub 프로젝트의 포스팅 단일 운영 문서다.
허브 글 작성, 하위 글 작성, 기존 글 수정, 이미지 alt 보강, 내부 링크 점검, 키워드 분리, frontmatter 설계, 품질 검수까지 모두 이 문서를 기준으로 실행한다.

삭제 예정 문서:
- `md-doc/now-posting-작업중.md`
- `md-doc/now-sujung.md`

위 두 문서의 실무 규칙은 이미 이 문서에 통합됐다.

## 0-B. Google 공식 참고 문서

글 작성, 수정, AdSense 재검토 대응 시 아래 Google 공식 문서를 우선 참고한다.
이 링크들은 "AI 티를 숨기는 방법"이 아니라, Google이 요구하는 사용자 가치·고유성·신뢰·대량 생성 방지 기준을 확인하기 위한 기준 문서다.

- AdSense 페이지 준비 기준:
  - https://support.google.com/adsense/answer/7299563
  - 적용 포인트: unique content, clear navigation, good user experience, original and relevant content.
- AdSense 참여 자격:
  - https://support.google.com/adsense/answer/9724
  - 적용 포인트: 사이트 소유권, 정책 준수, 본인 콘텐츠의 품질과 독창성.
- AdSense "site not ready" 대응:
  - https://support.google.com/adsense/answer/12176698
  - 적용 포인트: ad code/접근성/정책/콘텐츠 가치 문제를 콘텐츠 수정과 별도로 점검.
- Google Search helpful content:
  - https://developers.google.com/search/docs/fundamentals/creating-helpful-content
  - 적용 포인트: people-first content, original information, substantial value, Who/How/Why, E-E-A-T self-assessment.
- Google Search 생성형 AI 콘텐츠 가이드:
  - https://developers.google.com/search/docs/fundamentals/using-gen-ai-content
  - 적용 포인트: AI 사용 자체가 문제가 아니라, 사용자 가치 없이 대량 생성된 콘텐츠가 문제다.
- Google Search spam policies / scaled content abuse:
  - https://developers.google.com/search/docs/essentials/spam-policies#scaled-content-abuse
  - 적용 포인트: 검색 순위 조작 목적의 대량·비독창·저가치 페이지 생성 금지.

운영 해석:
- Google에는 선호 글자 수가 없다. KO 3,500자 / EN 900 words는 이 프로젝트의 내부 최소 하한선이다.
- sitemap에서 제외해도 공개 URL, 내부 링크, 과거 색인, 외부 링크를 통해 Google이 페이지를 발견할 수 있다.
- "AI 패턴 감지 회피"가 목표가 아니다. 목표는 사람이 읽어도 도움이 되는 고유 정보와 검증 가능한 기준을 넣는 것이다.
- 직접 경험이 없는 내용을 직접 경험처럼 쓰지 않는다. 대신 공개 정보, 실제 사용 조건, 관찰 가능한 기준, 실패 신호를 명확히 쓴다.

## 0-A. Critical Constraints

아래 규칙은 가장 우선순위가 높은 필수 규칙이다.
하나라도 누락되면 포스팅 미완료로 간주한다.

### [C1] Frontmatter Required

[DO]
- 모든 포스트는 `.mdx` 파일로 작성한다
- `slug`, `lang`, `category`, `title`, `description`, `publishedAt`, `readTime`, `headerGradient`, `pullQuote`, `hreflangSlug`를 반드시 채운다
- 파일명과 `slug`는 반드시 같아야 한다
- KO/EN 짝 파일은 같은 `hreflangSlug`와 같은 `publishedAt`를 사용한다

[NOT]
- 필수 frontmatter 누락 금지
- KO에서 `서울 명소 & 포토존` 사용 금지
- 파일명과 `slug` 불일치 금지

### [C2] Main Keyword Placement

[DO]
- 제목에 메인 키워드를 반드시 1회 포함한다
- 첫 문단에 메인 키워드 1회와 보조 키워드 1개 이상을 자연스럽게 포함한다
- H2 최소 1개 이상에 메인 키워드 또는 보조 키워드를 반영한다

[NOT]
- 메인 키워드 4회 초과 반복 금지
- 키워드만 나열하는 문장 금지
- 검색 의도 없는 추상 키워드 사용 금지

### [C3] Hub Backlink Required

[DO]
- 하위 글은 본문 어딘가에 허브 글로 가는 내부 링크를 반드시 포함한다
- 허브 링크는 자연스러운 설명형 앵커 텍스트로 연결한다

[NOT]
- Quick Summary나 Related panel에만 허브 링크를 두고 본문에서 누락하는 것 금지
- `여기`, `링크`, `click here` 같은 빈약한 앵커 텍스트 금지

### [C4] Locale-Isolated Internal Links

[DO]
- KO 글은 `/ko/hub/[slug]` 경로만 사용한다
- EN 글은 `/en/hub/[slug]` 경로만 사용한다
- 내부 링크는 실제 존재하는 글로만 연결한다

[NOT]
- KO 본문에서 EN 링크 사용 금지
- EN 본문에서 KO 링크 사용 금지
- 존재하지 않는 글로의 dead link 금지

### [C5] Completion Tracking Update

[DO]
- KO/EN 짝 글이 모두 끝난 뒤 `md-doc/kpop-blog-topics-300.md` 상태를 갱신한다
- 완료된 항목은 원래 형식을 유지한 채 `Sub [ ]`를 `Sub [v]`, `Hub`를 `Hub [v]`로 바꾼다
- 완료 후 `md-doc/kpop-blog-topics-300.md`의 topic 상태와 진행 로그를 함께 업데이트한다

[NOT]
- KO 또는 EN 한쪽만 완료된 상태에서 완료 체크 금지
- 원래 제목이나 번호 구조를 바꾸면서 체크하는 것 금지

### [C6] Structure, Fact Box, CTA

[DO]
- 장소형, 방문형, 촬영형 글은 본문 초반 H2 안에 실용 정보 bullet list를 반드시 넣는다
- 한 글 안에서 H2별 문단 수와 길이를 반드시 다르게 설계한다
- 2문단 고정 패턴은 구조 위반으로 보고, 초안 단계에서 반드시 깨뜨린다
- 장소형, 방문형, 촬영형 글은 본문 중간 이후 구간에서 mid-CTA 텍스트 링크를 반드시 검토한다
- 문맥상 자연스럽고 흐름을 해치지 않는다면 mid-CTA 1회 삽입을 기본값으로 본다
- mid-CTA는 보통 글의 50~70% 지점에서 1개만 두는 것을 기본값으로 본다
- 이 프로젝트는 추후 광고 삽입도 고려하므로, mid-CTA를 여러 개 넣지 않는다

[NOT]
- 모든 H2 아래를 2문단 고정 패턴으로 반복 금지
- 글 전체가 사실상 같은 문단 수 반복으로 읽히는 구조 금지
- 실용 정보 블록 생략 금지
- mid-CTA를 항상 0으로 처리하는 해석 금지
- mid-CTA를 "어색할 것 같으니 생략" 쪽으로 자동 해석하는 것 금지
- 모든 글에서 같은 위치, 같은 문장 템플릿으로 CTA 반복 금지

### [C7] Image Insertion and Reuse Ban

[DO]
- 글 작성 단계에서는 이미지 없이 본문, SEO, 내부 링크, alt 초안까지만 처리한다
- 본문 이미지 삽입은 항상 사용자가 마지막 단계에서 직접 처리한다
- Codex는 이미지가 필요한 글에 대해 "필요 이미지/시각 자료 brief"를 남긴다
- brief에는 이미지 위치 후보, 장면/표/비교 자료 종류, alt 초안, 재사용 금지 여부를 포함한다

[NOT]
- Codex가 본문 이미지 마크업을 넣는 행위 절대 금지
- 프로젝트 최상위 폴더와 모든 하위 폴더를 포함한 저장소 전체에서 기존에 존재하는 어떤 사진 파일도 글 작성용으로 사용하는 것 절대 금지
- 이전 포스팅에서 사용한 사진을 새 포스팅에 재사용하는 것 절대 금지
- 허브 글과 하위 글 사이에서 동일 사진을 재사용하는 것 절대 금지
- 같은 포스팅 안에 동일한 사진 파일을 두 번 이상 넣는 것 절대 금지

예외:
- 사용자가 현재 글의 이미지 삽입을 위해 직접 추가한 파일은 위 금지 대상에서 제외한다

### [C8] Content Depth — 내부 최소 품질 하한선

[DO]
- KO 글: frontmatter·##·-·!·<·빈 줄 제외 순수 본문 최소 3,500자
- 신규 KO 글은 가능한 경우 4,000자 이상을 기본 목표로 잡는다
- KO 글: 실질 정보가 충분하면 5,000자를 넘어도 된다 — 상한선 없음
- EN 글: 동일 기준 순수 본문 최소 900 words
- 신규 EN 글은 가능한 경우 1,100 words 이상을 기본 목표로 잡는다
- EN 글: 실질 정보가 충분하면 1,200 words를 넘어도 된다 — 상한선 없음
- 위 글자 수는 Google 공식 기준이 아니라 이 프로젝트의 내부 하한선이다
- 글자 수를 채우는 것만으로 AdSense 통과 기준을 충족했다고 판단하지 않는다
- 각 H2 섹션은 다른 섹션에 없는 새로운 정보를 반드시 포함한다
- 구체적 관찰, 계절·시간·피부타입별 차이, 실패 사례, 비교 기준처럼
  다른 사이트에서 찾기 어려운 내용을 섹션당 1개 이상 포함한다
- 글자수가 부족할 때는 정보 밀도를 높이는 방향으로만 채운다

[NOT]
- 글자수를 채우기 위해 같은 의미를 다른 표현으로 반복하는 것 금지
- 섹션 마지막에 해당 섹션 내용을 요약하는 문장 추가 금지
- 섹션 제목이나 불릿 내용을 바로 아래 산문에서 다시 풀어쓰는 것 금지
- 추상적 연결 문장으로 단락을 채우는 것 금지
- 템플릿형 마무리 표현 절대 금지:
  "결국", "따라서", "이런 이유로", "정리하자면", "핵심은",
  "중요한 것은", "마지막으로", "요약하면", "이 차이 때문에",
  "그래서 이 방식이", "결론적으로", "In conclusion", "Overall",
  "It is important to note", "To summarize", "Ultimately",
  "With that in mind", "Having said that"
- 4문장 이상 비슷한 길이의 문장 연속 금지
- 3문장 이상 같은 단어로 시작하는 문장 연속 금지
- "X는 Y보다 Z하다. A는 B보다 C하다." 평행 구조 문장 연속 2개 이상 금지

### [C9] Content Uniqueness — 글 간 본문 중복 방지

[DO]
- 글 작성 전 같은 허브 내 다른 글의 본문 논점을 먼저 확인한다
- 허브 글과 하위 글은 서로 다른 검색 의도와 다른 본문 내용을 담당한다
- 하위 글끼리도 설명하는 주요 논점과 예시가 달라야 한다
- KO와 EN은 같은 주제를 다루되 각 언어 독자의 검색 의도에 맞게 다르게 서술한다

[NOT]
- 허브 글에서 이미 설명한 본문 논점을 하위 글에서 반복 금지
- 하위 글 A의 본문 논점을 하위 글 B에서 거의 동일하게 다루는 것 금지
- 같은 예시, 같은 비교 구조, 같은 결론을 다른 글에서 재사용 금지
- KO 글의 본문을 그대로 번역해 EN 글로 사용하는 것 금지

### [C10] 대량 생성형 저품질 신호 방지 — AdSense "가치 없는 콘텐츠" 대응

이 섹션은 구글 애드센스가 "가치 없는 콘텐츠"로 거절할 수 있는 저품질 신호 목록이다.
Google은 AI 사용 자체를 금지하지 않는다.
문제는 사용자에게 새 가치가 없는 대량 생성형 문서, 반복 템플릿, 검증되지 않은 일반론이다.
아래 항목은 단순 권고가 아니라 **글 생성 시 위반 즉시 재작성 대상**이다.

---

#### [C10-1] "편이 ~합니다" 과다 반복 — 절대 금지

[NOT]
- "편이 좋습니다", "편이 낫습니다", "편이 안전합니다", "편이 실용적입니다",
  "편이 중요합니다", "편이 효율적입니다", "편이 편합니다" 등
  **"편이 ~합니다" 계열 표현을 글 전체에서 3회 이상 사용 금지**
- 같은 H2 섹션 안에서 "편이 ~합니다" 계열을 2회 이상 사용 금지

[DO]
- "~하는 것이 낫다", "~가 더 도움이 된다", "~쪽이 현실적이다",
  "~를 먼저 확인한다", "~가 더 잘 맞는다", "~로 시작하면 된다" 등
  다양한 서술 방식으로 대체한다
- 한 글에서 문장 끝 표현이 3회 이상 반복되는 패턴이 생기면 반드시 깬다

---

#### [C10-2] H2 전체를 "~할까" 의문형으로 통일 — 절대 금지

[NOT]
- 한 글의 H2 제목 중 절반 이상이 "~할까", "~일까", "~인가" 의문형으로 끝나는 것 금지
- 예시 위반 패턴:
  ```
  ## 무엇부터 줄이는 편이 좋을까
  ## 왜 진정 제품이 더 중요할까
  ## 어떻게 나누는 편이 덜 자극적일까
  ## 이유는 무엇일까
  ## 왜 일관성에 더 잘 반응할까
  ```
  → 이 패턴이 글 전체에 걸쳐 나타나면 대량 생성형 템플릿 신호로 보일 수 있다

[DO]
- H2 유형을 혼합한다: 의문형 / 서술형 / 명사형 / 조건형을 섞는다
- 예시 혼합:
  ```
  ## 피부가 민감할 때 먼저 줄여야 할 것
  ## 왜 진정 제품이 먼저인가
  ## 아침과 밤 루틴 분리 기준
  ## 계절이 바뀔 때 조정하는 방법
  ## 민감한 피부가 자주 하는 실수
  ```

---

#### [C10-3] 구체적 수치·성분·정보 없는 추상 서술 — 절대 금지

[NOT]
- 뷰티 글에서 성분명 없이 "진정 성분", "순한 보습", "기능성 레이어"만 쓰는 것 금지
- 장소 글에서 숫자 없이 "오전 첫 시간대", "붐비는 시간", "운영 체크" 수준으로만 쓰는 것 금지

## 5. Frontmatter 규칙

모든 포스트는 `.mdx` 파일로 작성하고 아래 frontmatter 구조를 따른다.

```mdx
---
slug: "korean-skincare-routine-guide"
lang: "en"
category: "K-Beauty & Hair"
title: "Complete Korean Skincare Routine Guide for Clear Skin"
description: "Build a practical Korean skincare routine with the right order, product roles, and beginner-friendly tips for clearer, more balanced skin."
publishedAt: "2026-03-15"
readTime: "6 Min Read"
headerGradient: "linear-gradient(135deg, #1a0a2e 0%, #f4258c 100%)"
pullQuote: "clear skin starts when the routine makes sense, not when the shelf gets crowded."
hreflangSlug: "korean-skincare-routine-guide"
---
```

필수 필드:
- `slug`
- `lang`
- `category`
- `title`
- `description`
- `publishedAt`
- `readTime`
- `headerGradient`
- `pullQuote`
- `hreflangSlug`

선택 필드:
- `nextSlug`
- `nextTitle`

필드 규칙:
- `lang`는 `ko` 또는 `en`
- KO category는 해당 토픽의 KO 카테고리명과 일치해야 한다
- EN category는 해당 토픽의 EN 카테고리명과 일치해야 한다
- category 값은 반드시 `md-doc/kpop-blog-topics-300.md`의 category 정의를 기준으로 한다
- 임의 번역 카테고리명 사용 금지
- `publishedAt`는 ISO 날짜 `YYYY-MM-DD`
- KO/EN 짝 파일은 `publishedAt`를 동일하게 맞춘다
- `title`은 고유해야 하며 60자 이하여야 한다
- `description`은 120~155자여야 한다
- `description` 첫 단어는 `title` 첫 단어와 같으면 안 된다
- `description`은 제목을 풀어쓰기 하지 말고 클릭 이유를 말해야 한다
- `pullQuote`는 기사 안의 한 문장으로, 인용부호 안에서 소문자로 시작한다
- `hreflangSlug`는 KO/EN 짝 글에서 동일해야 한다
- KO `readTime`은 자연스러운 한국어 형식으로 작성한다
- EN `readTime`은 `6 Min Read` 형식을 따른다

## 6. 작성 규칙

### 6-1. 인트로 역할

인트로는 3문장 이내로 아래 3가지 역할을 해야 한다.

1. 이 장소/주제를 정의한다
2. 사람들이 단순하게 이해하는 지점을 짚는다
3. 이 글이 무엇을 더 설명하는지 말한다

금지:
- `많은 사람은 X를 Y로 단순화하지만 실제로는`
- `Many people think of X as Y, but in reality`
- `~하는 사람은 보통 X 같지만, 실제로는` 및 모든 변형 — "실제로는"을 포함한 인트로 패턴 전면 금지
- `People often expect X, but` / `People usually think X, but` / `People usually want X, but` 변형 포함

같은 허브 안에서는 인트로 유형을 돌려 쓴다.

- A 질문형
- B 숫자/통계형
- C 장면 묘사형
- D 직설형

### 6-2. 본문 문체

- 설명 중심으로 쓴다
- 필요할 때만 리스트 사용
- 장식적인 문장보다 구체적인 장면, 동선, 비교 기준을 우선한다
- 과한 굵게, 과한 불릿, 세일즈 문체 금지

### 6-3. KO/EN 분리 원칙

- KO와 EN은 번역하지 않는다
- 같은 주제를 각 언어 검색 의도에 맞게 다시 쓴다

KO 초점:
- 왜 유명해졌는가
- 한국 사용자 기준 체감 차이
- 다른 서울 지역과 비교

EN 초점:
- 서울 맥락에서 왜 중요한가
- 외국인에게 낯선 배경 설명
- what it is / why it matters / how to read it

### 6-4. Pattern Diversification Rules

이 프로젝트는 여러 글이 한 묶음으로 발행되므로, 문서군 전체가 공장형 패턴으로 보이지 않게 구조를 분산해야 한다.
이 규칙은 선택형 미세 조정이 아니라, 검색 품질과 사람 손맛을 지키기 위한 최우선 구조 규칙이다.

필수 원칙:
- 모든 글의 H2 개수를 기계적으로 맞추지 않는다
- 글마다 문단 수와 문단 길이를 다르게 설계한다
- 같은 허브 안에서도 서론 길이, H2 수, 본문 호흡이 똑같아지지 않게 조정한다
- 같은 문장 길이와 같은 전개 리듬을 반복하지 않는다

H2 분산 원칙:
- H2는 글 성격에 따라 유연하게 구성한다
- 일반적으로 3~6개 범위에서 주제에 맞게 선택하되, 모든 글을 같은 개수로 맞추지 않는다
- 허브 글은 넓은 맥락 설명 중심, 하위 글은 구체 의도 해결 중심으로 H2 구성을 다르게 가져간다
- H2 언어는 검색자 관점에서 자연스럽게 읽히는 질문형, 비교형, 판단형을 우선 검토한다
- `…이유`, `…구간`, `…방식` 같은 편집 문어체를 기계적으로 반복하지 않는다

문단 분산 원칙:
- 각 H2 아래 문단 수를 동일하게 고정하지 않는다
- 어떤 섹션은 1~2문단으로 짧게 끝낼 수 있고, 어떤 섹션은 비교나 설명 때문에 더 길어질 수 있다
- 모든 섹션을 같은 줄 수, 같은 호흡으로 작성하는 패턴을 금지한다
- H2 산하 문단 길이를 기계적으로 2~3줄로 맞추지 않는다
- 어떤 H2는 짧고 임팩트 있게, 어떤 H2는 더 길고 설명적으로 써서 사람이 직접 다듬은 듯한 길이 차를 만든다
- 한 글 안에서 H2 아래 문단 수가 전부 같아지면 반드시 깨야 한다
- 2문단 고정 패턴이 3개 이상 연속되면 그 다음 H2는 반드시 1문단 또는 3문단 이상으로 작성한다
- H2가 5개인 글은 H2 2-3-4-5가 연속 2문단이 되지 않도록 하고, H2 3 또는 H2 5 중 하나는 반드시 1문단 또는 3문단 이상으로 설계한다
- 글 전체를 훑었을 때 "모든 H2가 2문단"처럼 보이면 초안 실패로 간주하고 다시 조정한다
- 분산은 장식이 아니라 필수다. 사람이 쓴 글처럼 짧게 치고 빠지는 섹션과 길게 설명하는 섹션이 같이 있어야 한다
- 왜 이 H2만 더 짧거나 더 긴지 설명할 수 없다면 다양화가 부족한 것으로 본다

아웃라인 분산 원칙:
- 허브 글과 하위 글은 전개 방식 자체를 다르게 설계한다
- 가능한 전개 유형 예시:
  - 비교형
  - 동선형
  - 해설형
  - 포토팁형
  - 시간대형
  - 입문 가이드형
- 같은 허브 안에서도 글마다 전개 유형을 달리해 반복 패턴을 줄인다

Quick Summary 분산 원칙:
- Quick Summary는 항상 3개 항목을 유지하되, 각 항목의 문장 길이와 리듬은 글마다 다르게 조정한다
- 요약 카드가 매번 같은 어순, 같은 길이, 같은 문장 템플릿으로 반복되지 않게 한다

금지:
- 모든 글을 같은 H2 개수로 맞추는 것
- 모든 H2 아래를 비슷한 길이의 문단 묶음으로 반복하는 것
- 허브 글과 하위 글이 사실상 같은 아웃라인을 공유하는 것
- Quick Summary 문장을 글마다 복사형 템플릿으로 쓰는 것

### 6-5. Practical Info Blocks

장소형 글, 방문형 글, 촬영형 글은 본문 초반에 실용 정보 블록을 반드시 넣는다.

필수 규칙:
- 본문 초반 H2 안에 bullet list 형식으로 배치한다
- `가는 법`, `추천 시간`, `운영시간`, `휴무일`, `요금`, `주의사항` 중 해당 주제에 맞는 항목을 1개 이상 반드시 포함한다
- 실용 정보는 에세이형 문장보다 방문 판단이나 촬영 판단에 도움이 되는 사실형 문장으로 쓴다
- KO와 EN 모두 동일하게 적용한다

주의:
- 모든 글의 첫 번째 H2에 기계적으로 넣지 않는다
- 모든 글에 같은 항목 조합을 반복하지 않는다
- 허브 글과 하위 글의 성격이 다르면 실용 정보의 위치와 형식도 다르게 가져간다

## 7. Google SEO 규칙

### 7-1. 메인 키워드

각 글은 언어별로 메인 키워드 1개를 가진다.

조건:
- 실제 검색창에 들어갈 법한 표현
- 추상적 테마 문구 금지
- 철학적, 감상적 표현 금지
- 같은 허브 안 다른 글과 중복 금지
- 가급적 롱테일 검색어 우선

매우 중요:
- 기존 문서나 과거 세션에 적혀 있던 제목을 고정값처럼 따르지 않는다
- `md-doc/kpop-blog-topics-300.md`의 제목과 메인 키워드는 기획 초안일 뿐 최종 고정값이 아니다
- 실제 글 작성 시에는 더 검색량이 높고 검색 의도가 선명한 실사용 키워드로 다시 설계한다
- 제목은 메인 키워드를 포함하되, 클릭 이유가 보이는 구체 효익이나 비교 포인트를 함께 붙인다

### 7-2. 보조 키워드

각 글은 보조 키워드 3~5개를 가진다.

역할:
- 메인 키워드 보강
- 세부 검색 의도 확장
- H2, 첫 문단, 중간 문단, 이미지 alt, 요약 카드에 분산 반영
- 필요 시 `예약`, `비용`, `프로필`, `촬영`, `코스`, `투어`, `추천 루트`처럼 전환 직전 검색 의도를 반영하는 보조 키워드도 검토한다

### 7-3. 배치 규칙

- `title`: 메인 키워드 포함
- 첫 문단: 메인 키워드 + 보조 키워드 1~2개 포함
- H2 1개 이상: 메인 키워드 또는 보조 키워드 포함
- 본문 중간 문단: 보조 키워드를 자연스럽게 나열하지 말고 맥락 안에 분산 반영
- 이미지 alt: 장면 설명 중심으로 보조 키워드 1개만 자연스럽게 포함 가능
- 마무리/요약: 메인 키워드 1회 자연스럽게 회수 가능

반복 제한:
- 메인 키워드는 본문에서 3~4회 이하
- 키워드만 나열하는 문장 금지
- 같은 문장 구조로 반복 삽입 금지

### 7-4. 제목과 설명 최적화

- 제목은 검색형 문구를 앞쪽에 두는 것을 우선 검토한다
- 제목에는 반드시 핵심 키워드가 들어가야 한다
- 제목은 과장보다 구체성, 비교성, 효익을 우선한다
- description은 제목 반복이 아니라 클릭 이유를 설명하는 독립 문장이어야 한다
- description에는 메인 키워드 또는 가장 가까운 변형 표현을 자연스럽게 포함할 수 있다

### 7-5. Google SEO 품질 원칙

- E-E-A-T 관점에서 구체 관찰, 현장성, 비교 기준을 넣는다
- 글 또는 사이트 차원에서 Who/How/Why가 설명되어야 한다
  - Who: 누가 운영·검수하는 콘텐츠인지
  - How: 어떤 기준으로 조사·작성·이미지 선정·검수했는지
  - Why: 검색자에게 어떤 실질 문제를 해결해 주기 위해 작성했는지
- 문단마다 정보 목적이 분명해야 한다
- 같은 의미를 다른 표현으로 반복해 길이만 늘리지 않는다
- 허브 글과 하위 글은 서로 다른 검색 의도를 담당하게 설계한다
- 내부 링크는 실제 존재하는 글만 연결한다
- 얇은 콘텐츠, 과장형 문구, 키워드 스터핑을 피한다

정보 밀도 기준 (AdSense 저품질 방지):
- 각 H2 섹션은 "왜", "언제", "어떻게", "무엇이 다른가" 중 하나에
  구체적으로 답해야 한다
- 장소글: 시간대별·계절별·혼잡도별 차이를 최소 1개 섹션에 포함한다
- 뷰티글: 피부타입별·계절별·초보자 실수 사례를 최소 1개 섹션에 포함한다
- "다른 사이트에서도 찾을 수 있는 일반론"만 있는 섹션은 삭제하거나
  구체적 관찰로 교체한다

대량 생성형 저품질 신호 방지:
- 문장 길이를 반드시 섞는다 (짧은 문장 10자 이내 + 긴 문장 40자 이상 혼재)
- 같은 형용사·부사를 기계적으로 반복하지 않는다
- 반복 점검 우선 대상: `좋습니다`, `중요합니다`, `경우가 많`, `편이`, `works best`, `often`, `usually`
- H2마다 문단 수가 달라야 한다 (1개, 2개, 3개, 4개 불균등 배분)
- 섹션 결론 단락이 섹션 제목의 재진술이면 삭제한다

### 7-6. AdSense 재검토 노출 규칙

- sitemap 제출 대상은 AdSense 재검토 표면을 정리하는 보조 수단일 뿐이다
- sitemap에서 제외된 공개 URL도 내부 링크, 과거 색인, 외부 링크를 통해 Google이 발견할 수 있다
- 얇은 글을 sitemap에서 뺀 것만으로 "품질 리스크가 사라졌다"고 판단하지 않는다
- 대표 심사 글은 길이, 고유 정보, 이미지/시각 자료, 내부 링크, 작성 기준이 모두 맞아야 한다
- 공개 상태로 남길 글은 장기적으로 최소 품질 하한선을 충족시키거나, 관련 허브/상위 글로 병합할지 검토한다
- noindex, 삭제, 병합, sitemap 제외 같은 색인 정책 변경은 사용자 승인 없이 적용하지 않는다
- 재심사 직전에는 `robots.txt`, `ads.txt`, `sitemap.xml`, 대표 hub/feed 노출 목록, AdSense head script 또는 verification meta를 함께 확인한다

## 8. 수정 규칙

기존 글을 수정할 때는 아래 순서로만 손댄다.

1. `title`
2. `description`
3. 첫 문단
4. H2 1~2개
5. 이미지 alt
6. 관련 글 패널 문구

기본 원칙:
- 본문을 갈아엎지 않는다
- 기존 논지와 구조는 최대한 유지한다
- 단, AdSense 저품질 대응, 얇은 콘텐츠 보강, 대량 생성형 반복 패턴 제거 작업에서는 본문 구조와 H2를 더 넓게 수정할 수 있다
- 이 경우에도 기존 글의 검색 의도와 내부 링크 관계는 유지한다
- 긴 문단만 선택적으로 2문단으로 나눈다
- 과한 리라이팅 금지

실무 한 줄:
- 제목은 더 검색 친화적으로
- 첫 문단은 키워드 더 선명하게
- H2는 1~2개만 검색형으로
- alt는 짧고 구체적으로

## 9. 허브와 하위 글 분리 규칙

이 프로젝트에서 가장 중요한 SEO 규칙 중 하나는 허브와 하위 글의 역할 분리다.

허브 글:
- 상위 검색 의도 담당
- 장소 전체의 상징성, 방문 흐름, 대표성, 비교 구조를 설명

하위 글:
- 구체 검색 의도 담당
- 포토존, 시간대, 케이블카, 카페, 야경 포인트 같은 세부 주제 담당

절대 금지:
- 허브가 하위 글 메인 키워드를 빼앗는 것
- 하위 글이 허브의 상위 의도를 다시 넓게 반복하는 것

예시:
- 허브: `남산 N서울타워 서울 야경 랜드마크`
- 하위1: `남산 N서울타워 야경 명소`
- 하위2: `남산 케이블카 스냅 사진`

허브에서는 하위 키워드를 메인처럼 반복하지 않는다.
하위 글 제목은 related panel이나 내부 링크에서만 세부 주제로 연결한다.

## 10. 내부 링크 규칙

- 모든 하위 글은 본문에 허브 글 역링크를 포함한다
- 본문 상단 또는 첫 번째 H2가 끝나는 지점 등, 문맥이 맞는 곳에 텍스트 기반 거미줄 내부 링크(In-content link)를 1~2개 자연스럽게 삽입한다
- 허브 글 `Related Guides` 또는 `함께 읽을 글`에는 실제 존재하는 하위 글만 넣는다
- 같은 허브 안에서 1~2개의 관련 글을 자연스럽게 교차 링크한다
- dead link 금지
- 내부 링크 anchor text는 목적지를 설명하는 자연어로 쓴다
- KO는 `/ko/hub/...`만 사용
- EN은 `/en/hub/...`만 사용
- Cross-language link 금지
- 같은 허브 강조 링크는 Quick Summary 아래의 dedicated panel로 정리한다

허브 완료 후 반드시:
- 허브 related panel 누락 점검
- 하위 글 역링크 점검
- 각 하위 글 related panel 누락 점검
- 키워드 중복 점검

## 11. 이미지 및 alt 규칙

### 11-1. 본문 이미지 문법

- 허브 글과 해당 하위 글 묶음이 모두 완성되기 전에는 본문 이미지 마크업을 먼저 넣지 않는다
- 글 작성 단계에서 Codex가 본문 이미지를 넣는 행위는 절대 금지한다
- 이미지 삽입 작업은 항상 사용자만 직접 한다
- Codex는 글 작성, 수정, SEO, 내부 링크, alt 초안까지만 담당하고 본문 이미지 마크업은 절대 추가하지 않는다
- Codex는 이미지 삽입이 필요한 글의 끝이나 작업 로그에 "필요 이미지/시각 자료 brief"를 남긴다
- brief는 실제 이미지 경로를 만들지 않고, 필요한 장면·비교표·단계표·alt 초안만 적는다
- 프로젝트 최상위 폴더와 모든 하위 폴더를 포함한 저장소 전체의 기존 사진 파일은 글 작성 과정에서 어떤 형태로도 사용하지 않는다
- 단, 사용자가 현재 글의 이미지 삽입을 위해 직접 추가한 파일은 예외로 보고 사용자가 직접 삽입할 수 있다
- 이미지 삽입은 허브와 하위 글의 본문, 내부 링크, related panel 정리가 끝난 뒤 마지막 단계에서만 사용자가 처리한다
- 이미지를 넣는 시점에는 각 H2 바로 아래 첫 줄에만 `![alt](/images/...webp)`를 반영한다
- 실제 `.webp` 파일이 준비되기 전에는 깨진 `/images/...` 경로를 본문에 남기지 않는다
- raw HTML `<img>` 금지

### 11-2. 이미지 경로

- 저장 위치: `public/images/hub/[hub-slug]/`
- MDX 경로: `/images/hub/[hub-slug]/[file]-kr.webp` 또는 `-en.webp`
- 최종 경로는 항상 `.webp`
- KO 파일 이미지는 `-kr.webp`, EN 파일 이미지는 `-en.webp`로 끝난다
- 사용자가 원본 확장자를 생략해도 실제 원본을 확인한 뒤 게시본은 항상 `.webp`로 만든다
- 원본이 `.png`, `.jpg`, `.jpeg`여도 본문 삽입 전 `.webp`로 변환한다
- 글당 이미지 수는 보통 3~7개를 기준으로 한다
- 이전 포스팅에서 사용한 사진을 새 포스팅에 다시 쓰는 것은 절대 금지한다
- 같은 허브 글과 하위 글 사이에서도 동일 사진 재사용을 절대 금지한다
- 같은 포스팅 안에 동일한 사진 파일을 두 번 이상 넣는 것도 절대 금지한다

### 11-3. alt 규칙

- alt는 비워두지 않는다
- 장면 설명 우선
- 검색 의도 키워드 1개만 자연스럽게 포함 가능
- 짧고 구체적으로 쓴다
- 시적인 표현 금지
- 추상적 감상 금지
- 키워드 나열 금지
- 반복 패턴 금지
- 파일명은 사용자가 채워도 되지만 alt는 Codex가 기본적으로 작성한다

좋은 alt:
- 실제 장면이 바로 떠오른다
- 짧다
- 키워드 1개가 자연스럽다

나쁜 alt:
- 추상적이다
- 문학적이다
- 검색어 덩어리처럼 보인다

### 11-4. alt 수정 순서

1. 허브와 하위 글 묶음이 모두 완성됐는지 먼저 확인
2. 실제 `.webp` 경로 확인

4. 각 H2 바로 아래에만 이미지 마크업 반영

## 13. CTA 규칙

기본 규칙:
- 하단 배너 CTA는 필수
- 링크는 `/ko` 또는 `/en`

soft conversion bridge:
- 본문 중간에 1~2회만 제한적으로 허용 가능
- 섹션 문맥과 정확히 맞아야 한다
- 공격적 판매 문구 금지
- 같은 문구 반복 금지
- 최종 CTA는 여전히 하단 배너가 중심

Mid-CTA 검토 규칙 (본문 중간 전환 링크):
- 장소형, 방문형, 촬영형 글에서는 mid-CTA 텍스트 링크를 반드시 검토한다
- 문맥상 어색하지 않다면 본문 중간에 1회 넣는 것을 기본값으로 본다
- 기본 개수는 글당 최대 1개다
- 위치는 보통 전체 글의 50~70% 지점을 우선 검토한다
- 광고나 다른 전환 요소가 함께 들어갈 수 있으므로 mid-CTA를 여러 개 중첩하지 않는다
- 사용자 흐름상 자연스러운 연결이 가능한 글이면 먼저 삽입 가능 지점을 찾는다
- mid-CTA는 결제 유도 문구를 노골적으로 밀어붙이는 것이 아니라, 본문 주제에서 결과물/서비스로 자연스럽게 넘어가는 한 줄 연결이어야 한다
- AdSense 재검토 대응 중에는 판매성 문구처럼 보이면 mid-CTA를 생략하고 작업 로그에 생략 이유를 남긴다
- 위치는 전체 글의 중간 이후 구간을 우선 검토하되, 모든 글에서 같은 위치로 고정하지 않는다
- Quick Summary 직전 고정 삽입 금지
- 같은 문장 템플릿을 여러 글에서 반복하지 않는다
- 링크 문구는 해당 글 주제와 자연스럽게 이어져야 하며, 과한 판매 문구처럼 보이지 않아야 한다
- KO와 EN 모두 동일하게 적용한다

금지:
- 모든 H2마다 CTA 삽입
- mid-CTA를 검토 없이 0개로 두는 것
- mid-CTA를 2개 이상 반복 삽입하는 것
- `/create` 직접 유도 문구를 기본 규칙처럼 쓰는 것
- 본문을 랜딩 페이지처럼 바꾸는 것

## 15. 품질 게이트

최종 제출 전 반드시 점검:

콘텐츠:
- title 60자 이하
- description 120~155자
- description이 title 반복이 아닌가
- 인트로 3역할 충족
- 같은 허브 내 인트로 유형 중복 없는가
- 패딩 문장 없는가

AdSense 콘텐츠 품질 체크 (제출 전 필수):
- KO 순수 본문 3,500자 이상인가 (frontmatter·##·-·!·< 제외)
- EN 순수 본문 900 words 이상인가
- 글자 수 기준을 "통과 보장"으로 오해하지 않았는가
- 각 H2 섹션이 다른 섹션에 없는 새 정보를 담고 있는가
- 같은 허브 내 다른 글과 본문 논점이 중복되지 않는가
- 템플릿형 마무리 표현("결국", "따라서", "정리하자면" 등)이 없는가
- H2마다 문단 수가 균일하지 않은가 (2문단 고정 패턴 금지)
- 섹션 결론 단락이 섹션 제목을 단순 재진술하지 않는가
- 4문장 이상 비슷한 길이의 문장이 연속되지 않는가

[C10] 대량 생성형 저품질 신호 체크 (애드센스 거절 원인 — 반드시 확인):
- "편이 ~합니다" 계열 표현이 글 전체에서 2회 이하인가
- H2 제목이 의문형("~할까")으로 절반 이상 통일되어 있지 않은가
- 뷰티 글이라면 성분명(센텔라, 나이아신아마이드, 히알루론산 등)이 최소 3개 이상 있는가
- 장소 글이라면 숫자 정보(시간·요금·출구·도보 분 등)가 최소 3개 이상 있는가
- 직접 경험이 있는 글이라면 구체 관찰이 최소 1개 이상인가
- 직접 경험이 없는 글이라면 1인칭 경험 표현 없이 검증 가능한 선택 기준을 충분히 넣었는가
- "~경우가 많다" 표현이 글 전체에서 4회 이하인가
- "누가/어떻게/왜 만들었는지"가 사이트 소개 또는 글의 품질과 모순되지 않는가
- 공개 글이 sitemap에서 빠졌더라도 내부 링크로 계속 강하게 노출되는 얇은 글이 아닌가

키워드:
- 메인 키워드가 title과 첫 문단에 있는가
- 보조 키워드가 본문에 맥락상 자연스럽게 분산되어 있는가
- 같은 허브 안 다른 글과 키워드 겹치지 않는가

구조:
- H2 아래 문단 수가 전부 같지 않은가
- 2문단 고정 패턴이 글 전체를 지배하지 않는가
- 글 전체를 훑었을 때 사람 손으로 다듬은 듯한 길이 차가 보이는가
- H2 문장이 검색자 문장처럼 자연스럽게 읽히는가

이미지:
- body image alt 비어 있지 않은가
- alt가 짧고 구체적인가
- `.webp` 경로인가
- Codex 작성 단계라면 실제 이미지 마크업 대신 필요한 이미지/시각 자료 brief를 남겼는가
- 사용자가 이미지를 삽입한 뒤에는 깨진 경로, 중복 이미지, 재사용 이미지가 없는가

링크:
- 하위 글이면 허브 역링크가 본문에 있는가
- dead link 없는가
- 언어 혼합 링크 없는가
- 본문 상단 또는 첫 번째 H2 이후에 문맥형 내부 링크 1~2개를 둘 수 있는 글이라면 자연스럽게 반영했는가

CTA:
- 하단 배너 유지됐는가
- 링크가 `/ko` 또는 `/en`인가
- 장소형, 방문형, 촬영형 글이라면 본문 초반 실용 정보 bullet list가 있는가
- 장소형, 방문형, 촬영형 글이라면 mid-CTA를 검토했고 문맥상 자연스러운 경우 1회 반영했는가
- 사용자 흐름상 자연스러운 연결이 가능한 글인데도 mid-CTA를 아무 근거 없이 생략하지 않았는가
- AdSense 재검토 대응 중 판매성 문구처럼 보이는 mid-CTA는 생략하고 이유를 남겼는가

법적:
- 금지된 인물/그룹/기획사/저작권 표현이 없는가
- 특정 브랜드와 인물을 연결하는 표현이 없는가

## 20. 수정 작업용 한 줄 프롬프트

기존 글 수정 시에는 아래 기준으로 본다.

- 메인 키워드는 유지
- title은 더 검색 친화적으로 미세 조정
- description은 클릭 이유 중심으로 재작성
- 첫 문단에 메인 키워드와 보조 키워드 1~2개 반영
- H2는 1~2개만 더 검색형으로 조정
- alt는 짧고 구체적으로 보정
- 본문 나머지 구조와 논지는 유지
