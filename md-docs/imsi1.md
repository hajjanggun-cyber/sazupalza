[문서 통합 검토: post-codex-command.md vs haha_flow.md]

[요약 의견]

- 최종적으로 하나로 합친다면, 기준 문서는 `post-codex-command.md`가 더 적합함.
- 이유는 `post-codex-command.md`가 작업 시작 순서, 필수 제약, SEO, frontmatter, 완료 규칙까지 포함한 "마스터 운영 문서" 구조이기 때문임.
- 반면 `haha_flow.md`는 현재 블로그 포스트 생산 규칙을 짧게 압축한 "실무 요약본"에 가까움.
- 따라서 `haha_flow.md`를 메인으로 삼기보다, `post-codex-command.md`를 메인으로 두고 현재 프로젝트에 맞게 축약/치환하는 편이 안정적임.

[비교 결과]

1. `post-codex-command.md`의 장점

- 작업 우선순위와 시작 순서가 명확함
- critical constraints가 분리돼 있어 실수 방지에 좋음
- frontmatter, SEO, internal link, completion update까지 end-to-end 흐름이 있음
- "무엇을 읽고 시작해야 하는지"까지 적혀 있어 운영 문서로 적합함

2. `post-codex-command.md`의 약점

- 현재 프로젝트(`sajupalza`) 기준으로는 과한 규칙이 섞여 있음
- 예전 프로젝트 경로, 파일명, 라우트, frontmatter 방식이 남아 있음
- 현 구조는 `.mdx` 기반 문서가 아니라 `lib/blog/*-posts.ts` 데이터 구조인데, 문서가 이를 반영하지 못함
- locale 규칙도 현재 실제 렌더링 방식과 일부 다름

3. `haha_flow.md`의 장점

- 현재 프로젝트 구조에 더 가까움
- `slug`, `seoTitle`, `description`, `keywords`, `toc`, `faq` 중심으로 실제 작업 단위가 맞아 있음
- "slug와 주제가 일치해야 한다", "키워드 중복 피해야 한다" 같은 현재 핵심 문제에 더 직접적임
- 실무자가 빠르게 보고 작업하기엔 더 가벼움

4. `haha_flow.md`의 약점

- 작업 흐름 문서로는 부족함
- 시작 전 확인, 완료 후 업데이트, 예외 처리, 파일 단위 체크 순서가 약함
- relatedPosts, 앵커 텍스트, 운영 문서 업데이트 같은 후속 작업 규칙이 `post-codex-command.md`보다 덜 체계적임
- "왜 이 규칙이 필요한지"보다 항목 나열에 가까워 확장성이 낮음

[통합 방향 제안]

- 메인 파일명은 `post-codex-command.md` 쪽이 더 나아 보임
- 다만 제목은 현재 프로젝트명에 맞게 바꾸는 것이 좋음
  - 예: `Sajupalza Unified Post Command`
  - 또는 `Sajupalza Blog Writing Command`

통합 방식:
- `post-codex-command.md`의 문서 골격 유지
- 아래 항목은 `haha_flow.md` 기준으로 덮어쓰기
  - 프로젝트 경로
  - 실제 데이터 저장 위치
  - 포스트 필수 필드
  - slug/seoTitle/description/keywords/toc/faq 규칙
  - 현재 SEO 위험 사례

[실제로 고쳐야 할 것]

1. 파일 포맷 기준 수정
- `.mdx` 작성 기준 제거 또는 별도 부록으로 분리
- 현재 기준인 `lib/blog/*-posts.ts` 구조로 교체

2. 경로/라우트 전면 교체
- `kstyleshot`, `/ko/hub/[slug]`, `content/hub/...` 같은 예전 구조 삭제
- 현재 사이트 라우트와 파일 구조 기준으로 재작성

3. locale 링크 규칙 현실화
- 현재 렌더러가 내부 링크를 locale 기준으로 정규화하므로, "소스에서 반드시 /ko /en만 써야 한다"는 식의 강제 문구는 완화 필요
- 대신 "최종 의도는 locale 일관성"으로 쓰는 편이 현실적임

4. 완료 체크 규칙 현행화
- 예전 토픽 관리 문서 참조 규칙 삭제
- 대신 `md-docs/now-posting.md`, `md-docs/seo-residual-checklist.md` 같은 현재 운영 문서 기준으로 정리

- 이 부분은 `haha_flow.md`보다 더 명시적으로 넣는 것이 좋음
- 현재 SEO에서 실제로 중요한 잔여 이슈이기 때문임
