# 제미나이 분석에 대한 정밀 검토 의견

검토일: 2026-05-03 KST
대상 파일: `md-docs/post-codex-command.md`
관련 보조 파일: `md-docs/post-codex-lite.md`
검토 결론: 제미나이의 3개 지적 중 1번과 2번은 매우 중요하고 실제 수정해야 한다. 3번은 방향은 맞지만, 제시된 중복 사례는 메인 포스트 `title` 중복이라기보다 `relatedPosts.title` 및 본문 링크 문구 반복에 가깝다. 그래도 기본 `title` 고유성 규칙은 반드시 강화하는 편이 맞다.

참고한 공식 기준:
- Google Search Central `Creating helpful, reliable, people-first content`: 원본성, 충분한 가치, 대량 생산 느낌, 특정 글자 수 집착 금지, people-first 목적을 강조한다.
- Google Search spam policies `Scaled content abuse`: 많은 페이지를 검색 순위 조작 목적으로 만들고 사용자 가치가 낮으면 생성 방식과 무관하게 문제가 될 수 있다.
- Google AdSense Help `site isn't ready to show ads`: unique content와 좋은 사용자 경험 부족은 승인/광고 준비 문제로 이어질 수 있다.

## 1. 최종 판단

제미나이 의견은 이전 오프스 의견보다 한 단계 더 실무적으로 맞다. 특히 내가 앞서 `post-codex-command.md`에 반복 방지 규칙을 추가했지만, 아직 문서 안에 AI가 그대로 따라 하기 쉬운 구조가 남아 있다.

가장 중요한 판단은 이것이다.

1. `Series H2 Template Rules`의 코드블록은 삭제하거나 비복사형 의도 프롬프트로 바꾸는 것이 맞다.
2. "무조건 신규 글 우선" 흐름은 얇은 시리즈 글을 계속 만들 위험이 있으므로, 개별 발행과 통합 발행을 가르는 게이트를 추가해야 한다.
3. `seoTitle`뿐 아니라 기본 `title`도 고유해야 한다는 규칙을 강화해야 한다. 단, 제미나이가 든 "13개 같은 title" 사례는 대부분 메인 글 title이 아니라 `relatedPosts.title` 반복이다.
4. `post-codex-command.md`만 고치면 부족하다. 실제 작업자가 `post-codex-lite.md`를 쓰면 같은 문제가 재발한다. 둘을 같이 맞춰야 한다.

## 2. 제미나이 1번: 시리즈 H2 템플릿은 독이 든 성배인가?

판단: 맞다. 가장 우선순위가 높다.

현재 `post-codex-command.md`에는 이미 "복사 가능한 제목 목록이 아니라 정보 단위"라고 경고가 들어갔다. 하지만 바로 아래에 다음처럼 명확한 코드블록이 남아 있다.

- `H2-1: [일간명]이란? — 오행·음양·기본 에너지`
- `H2-2: [일간명] 성격 특징 — 장점과 단점`
- `H2-3: [일간명] 직업 적성 — 천직과 피해야 할 분야`
- `H2-4: [일간명] 연애·결혼 스타일`
- `H2-5: [일간명] 2026 병오년 운세 흐름`

AI는 보통 추상 경고보다 구체 템플릿을 더 강하게 따른다. 그래서 경고문을 아무리 위에 써도, 아래 코드블록이 남아 있으면 실제 출력은 다시 템플릿 복사로 돌아갈 가능성이 높다.

특히 `post-codex-lite.md`는 더 위험하다. lite 파일에는 아직 "시리즈 H2 템플릿"이 짧고 선명한 목록 형태로 남아 있다. 실제 작업자가 lite를 보면 command 파일의 보강 규칙을 우회할 수 있다.

수정해도 되는가: 해야 한다.

수정 방식:

- H2 코드블록을 삭제한다.
- "시리즈별로 반드시 다룰 정보 단위"만 남긴다.
- 각 정보 단위는 H2 제목이 아니라 질문/판단 과제로 작성한다.
- 예: `기본 에너지` 대신 "이 일간/일주가 다른 글과 구분되는 가장 큰 현실 문제는 무엇인가?"
- 예: `성격 특징` 대신 "이 성향이 실제 관계, 직업, 돈, 건강 중 어디서 가장 자주 문제로 드러나는가?"
- 예: `2026 운세` 대신 "병오년이 이 구조에 주는 고유한 자극이 합/충/상생/상극 중 무엇이며, 왜 다른 일간과 다른가?"

하면 안 되는 방식:

- 코드블록 위에 경고문만 더 추가하는 것. 이미 했지만 충분하지 않다.
- `H2-1`, `H2-2` 형식을 유지하는 것. 번호가 있으면 AI가 순서를 고정한다.
- "변주하라"는 말만 남기고 실제 복사 가능한 문장을 유지하는 것.

권장 결론:

`post-codex-command.md`와 `post-codex-lite.md`의 시리즈 H2 템플릿은 둘 다 비복사형 프롬프트로 바꾸는 것이 맞다. 이건 해도 되는 수준이 아니라 해야 하는 수정이다.

## 3. 제미나이 2번: 통합 발행 원칙 부재

판단: 매우 중요하다. 다만 "긴 글 1개를 애드센스가 선호한다"는 표현은 정확히는 조심해야 한다.

Google 공식 문서의 핵심은 "긴 글" 자체가 아니다. Google은 특정 글자 수를 선호하지 않는다고 명시한다. 중요한 것은 독창성, 실질적 가치, 사람들이 다시 검색하지 않아도 될 만큼의 만족도, 대량 생산처럼 보이지 않는 관리 품질이다.

따라서 제미나이의 방향은 맞지만, 문서에는 이렇게 넣어야 한다.

잘못된 표현:

- "애드센스는 긴 글 1개를 선호한다."
- "2,000자를 못 채우면 통합하라."
- "시리즈 글은 만들지 말라."

정확한 표현:

- "개별 글이 독립 검색 의도와 독창적 실전 해석을 충분히 갖지 못하면 통합한다."
- "길이보다 정보 밀도와 중복 없는 가치가 우선이다."
- "각 일간/일주/오행 글이 서로 다른 문제, 사례, 한계, 판단 기준을 갖지 못하면 pillar content로 묶는다."

현재 `post-codex-command.md`의 문제:

- `0-AA-3. 신규 글 우선순위 원칙`이 "검색형 신규 글 발행"을 강하게 밀고 있다.
- `14. Series Cannibalization Prevention Rules`는 시리즈 중복 방지를 말하지만, "언제 시리즈를 포기하고 통합해야 하는가"는 약하다.
- `19. Repetition Cleanup Strategy`는 기존 반복 정비에는 좋지만, 신규 글을 만들기 전 분리/통합 결정을 강제하지 않는다.

수정해도 되는가: 해야 한다.

추가해야 할 규칙:

- `Split vs Consolidate Gate` 또는 `통합 발행 판단 규칙` 신설
- 개별 포스트로 분리하려면 아래 조건을 통과해야 한다:
  - 별도 검색 의도가 명확한가?
  - 기존 글과 다른 H2 질문을 4개 이상 만들 수 있는가?
  - 해당 글만의 실전 예시, 오해, 한계, 비교 기준이 있는가?
  - title/seoTitle/description이 기존 글과 후위어만 다른 수준이 아닌가?
  - KO/EN 모두 독립적으로 쓸 가치가 있는가?
- 통과하지 못하면:
  - 통합형 pillar 글로 작성
  - TOC로 세부 항목 점프 제공
  - 이후 Search Console에서 특정 하위 쿼리가 충분히 분리될 때만 개별 글로 확장

적용 예시:

- 60갑자 전체를 무조건 각각 1글로 만드는 것은 위험하다.
- 하지만 갑오, 경자처럼 2026 병오년과 충/동기/압박이 뚜렷하고 독립 쿼리가 있는 글은 개별 발행 가능하다.
- 차별 포인트가 약한 일주는 "2026 병오년 일주별 운세 통합표" 같은 pillar 글 안의 섹션으로 먼저 다루는 편이 낫다.

권장 결론:

개별 시리즈 발행 자체를 금지할 필요는 없다. 하지만 "개별 발행 자격 심사"가 반드시 필요하다. 이 규칙은 `post-codex-command.md`와 `post-codex-lite.md` 둘 다 넣어야 한다.

## 4. 제미나이 3번: seoTitle만 강조하고 기본 title 고유성을 놓쳤는가?

판단: 방향은 맞다. 하지만 제시된 근거 예시는 부정확하다.

내가 확인한 실제 데이터:

- `사주 일주 완전정복: 나의 본질과 배우자 복을 읽는 법`은 전체 `title:` 속성 검색에서 13회 나온다.
- 그러나 이 13회 대부분은 메인 포스트의 기본 `title`이 아니라 `relatedPosts.title` 또는 본문 내부 링크 문구다.
- 실제 `day-pillar` 메인 포스트는 `slug: 'day-pillar'` 아래에 하나 있고, 나머지는 해당 글을 가리키는 관련글 제목 반복이다.

따라서 "13개의 글이 같은 메인 title을 가진다"는 말은 현재 확인 기준으로는 맞지 않다.

그래도 수정해야 하는 이유:

- `title`은 포스트 필수 필드다.
- 현재 렌더러는 H1과 메타에 주로 `seoTitle`을 쓰지만, `title`은 fallback, 관련 목록, 내부 참조, OG 이미지, 데이터 정합성에 영향을 줄 수 있다.
- `post-codex-command.md` 2장에서는 `title`을 필수 필드로만 나열하고, 고유성 검사는 약하다.
- 5-1 제목 규칙은 `seoTitle` 중심으로 쓰여 있어 기본 `title`까지 같은 강도로 관리한다는 압력이 부족하다.

수정해도 되는가: 해야 한다.

추가해야 할 규칙:

- 기본 `title`도 전체 포스트의 메인 title 기준으로 고유해야 한다.
- `title`, `seoTitle`, `seoTitleEn`은 모두 중복 검사를 한다.
- 같은 카테고리에서는 `title`의 공통 접두어/후위어 반복을 제한한다.
- `title`은 가능하면 해당 글의 핵심 키워드나 고유 차별 포인트가 앞쪽에 와야 한다.
- `relatedPosts.title`은 같은 대상 글을 여러 번 가리키면 반복될 수 있으므로, 이것을 메인 title 중복으로 오판하지 않는다. 다만 관련글 앵커 문구가 너무 오래된 제목이면 최신 canonical title과 맞춰 정리한다.

하면 안 되는 방식:

- `relatedPosts.title` 반복을 근거로 모든 메인 title을 대량 변경하는 것.
- Search Console 데이터 없이 slug까지 바꾸는 것.
- `title`과 `seoTitle`을 무조건 완전히 다르게 만들려는 것. 둘은 같은 검색 의도를 공유하되, 화면/메타 역할에 맞게 표현만 조정하면 된다.

권장 결론:

제미나이의 "title도 관리하라"는 방향은 맞다. 하지만 13개 중복 사례는 relatedPosts 반복이므로 진단 문구는 고쳐야 한다. 수정 시에는 `main post title`과 `relatedPosts.title`을 구분하는 규칙까지 같이 넣는 것이 정확하다.

## 5. post-codex-lite.md도 같이 수정해야 하는가?

판단: 해야 한다.

이유:

- `post-codex-lite.md`는 짧은 작업 지침이라 실제 글 작성 때 더 자주 사용될 가능성이 있다.
- 현재 lite 파일에는 H2 템플릿이 더 짧고 복사하기 쉬운 형태로 남아 있다.
- command 파일만 고치고 lite를 놔두면, 작업자는 더 쉬운 lite를 따라가고 같은 문제가 반복된다.

수정 범위:

- lite의 `시리즈 H2 템플릿`을 삭제하거나 비복사형 정보 단위로 교체
- `Split vs Consolidate Gate` 추가
- 기본 `title` 고유성 체크 추가
- description 분량 표현 금지, CTA 반복 금지, CSS 변경 금지 규칙은 이미 command에 들어간 내용을 lite에도 압축 반영

## 6. 하면 되는 수정 / 하지 말아야 할 수정

### 해야 하는 수정

1. `post-codex-command.md`의 H2 코드블록 템플릿 제거 또는 비복사형 의도 프롬프트화
2. `post-codex-lite.md`의 H2 템플릿도 동일하게 제거 또는 교체
3. 신규 글 작성 전에 `개별 발행 vs 통합 발행` 판단 게이트 추가
4. 기본 `title` 고유성 규칙 강화
5. `relatedPosts.title` 반복과 메인 포스트 `title` 중복을 구분하는 검사 기준 추가
6. "긴 글이라서 좋은 것"이 아니라 "독창적이고 충분한 가치가 있어서 좋은 것"이라고 문구 정리
7. 기존 글 정비 시 Search Console, 핵심 URL, description 반복, 제목 후위어 반복 순서로 우선순위화

### 안 해도 되는 수정

1. CSS 변경
2. relatedPosts가 같은 대상 글을 여러 번 가리킨다는 이유만으로 모든 related title을 즉시 대량 변경
3. 60갑자/일간/오행 시리즈 자체를 전면 금지
4. 모든 글을 무조건 pillar content로 통합
5. 기존 slug 대량 변경
6. "2,000자 미만이면 무조건 통합" 같은 기계적 기준 추가

### 하면 안 되는 수정

1. H2 템플릿 위에 경고문만 더 붙이고 코드블록은 그대로 두는 것
2. `완벽`, `총정리`, `3,000자` 같은 단어만 지우고 본문 정보 밀도는 그대로 두는 표면 수정
3. 구글이 특정 글자 수를 선호한다고 문서에 쓰는 것
4. 애드센스가 "긴 글 1개"를 선호한다고 단정하는 것
5. 관련글 반복을 메인 title 중복으로 오진해서 무리하게 전체 수정하는 것

## 7. 내 최종 의견

제미나이 분석은 실행 관점에서 상당히 날카롭다. 특히 "문서 안의 명확한 템플릿이 AI에게 복붙 신호를 준다"는 지적은 맞다. 내가 앞서 추가한 반복 방지 규칙은 방향은 맞지만, H2 코드블록을 남겨 둔 상태라 재발 가능성이 있다.

따라서 다음 수정은 하는 것이 맞다.

1. `post-codex-command.md`와 `post-codex-lite.md`에서 시리즈 H2 템플릿을 비복사형 의도 프롬프트로 교체
2. `Split vs Consolidate Gate` 추가
3. 기본 `title` 고유성 규칙 추가
4. 관련글 title 반복과 메인 title 중복을 구분하는 검사 규칙 추가

현재 `description` 규칙은 line 530~535에 있다. 내용은 "실제 다루는 범위 요약", "제목과 다른 내용 약속 금지", "키워드 목록처럼 나열 금지" 정도다. 하지만 `3,000자`, `2,500자`, `전문 해설`, `전문 데이터` 같은 분량/품질 과시형 문구를 금지하지 않는다.

실제 글 데이터에서도 문제가 확인된다.

- `3,000자` 검색 결과: 전체 `lib/blog/*.ts`에서 54회

### 2-4. CTA/관련글 구조 반복 문제도 실제로 있다

## 3. 오프스 분석 중 보정이 필요한 부분

### 3-1. "실제 글이 전혀 따르지 않는다"는 표현은 과하다

오래된 글에는 오프스 지적이 강하게 맞다. `3,000자`, inline gradient CTA, `완벽 가이드`, `완전정복`, "Conclusion" 패턴이 실제로 많이 남아 있다.

하지만 최근 추가된 33~36번 글은 예전 글보다는 낫다.

- `saju-unknown-birth-time`, `daewoon-change`, `day-branch-reading`, `ilgan-career-table`은 표, 체크리스트, 한계 설명, 주제 분리를 꽤 반영하고 있다.
- H2도 30~32번 일주 글처럼 완전히 동일한 5개 템플릿만 반복하지는 않는다.

그래도 남은 문제는 있다.

- CTA 문구가 거의 동일하다.
- 결론 문단이 `결론적으로` / `In short` 패턴으로 반복된다.
- 중간 CTA 위치가 거의 고정되어 있다.
- 일부 EN 도입부는 `This guide...`, `This article...` 패턴이 반복된다.

정확한 표현은 "전체 누적 글에는 규칙 위반이 많고, 최근 글도 일부 반복 패턴이 남아 있다"가 맞다.

### 3-2. "HTML 메타 구조를 픽셀 단위로 변형"은 권장하지 않는다

오프스가 말한 "관련글 박스가 픽셀 단위로 동일"은 문제의 핵심이 아니다. 일관된 UI 컴포넌트는 오히려 유지보수와 UX 측면에서 맞다.

수정 방향은 아래가 더 정확하다.

- 같은 inline style을 글마다 박는 방식은 줄인다.
- 가능하면 `related-posts-inline`, `cta-box` 같은 공통 class나 렌더러 컴포넌트로 통일한다.
- 다만 글마다 CTA의 목적, 문구, 위치, 앞뒤 문맥은 바꾼다.
- 관련글 박스도 "함께 보면 좋은 글"만 반복하지 말고 "먼저 읽을 글", "다음 단계", "비교해서 볼 글", "이론 배경" 등 역할을 나눈다.

즉, CSS 랜덤화가 아니라 "콘텐츠 구조의 기능적 변주"가 필요하다.

추가 권장 위치:

추가해야 할 체크:

- H2 제목 자체는 매 글마다 고유하게 바꾸도록 강제한다.
- 같은 시리즈에서 H2 순서도 필요하면 바꿀 수 있게 한다.
- 예: 일주 글이라도 어떤 글은 2026 이슈를 앞에 두고, 어떤 글은 연애 패턴을 먼저 두고, 어떤 글은 직업·재물 리스크를 중심으로 잡을 수 있어야 한다.

3. inline gradient CTA가 남아 있는 오래된 글
4. `Conclusion` H2를 반복하는 EN 글
5. Search Console 노출은 있는데 CTR 낮은 글

## 5. 실제 글 기준으로 본 문제 우선순위

### 최우선: description의 분량/과시 문구 제거

이건 문서 수정과 실제 글 수정 모두 필요하다. `description`은 검색 결과 스니펫 후보가 될 수 있으므로, "3,000자 전문 해설" 같은 문구는 품질 신호보다 광고 문구처럼 보일 가능성이 크다.

우선순위가 높은 이유:

- 검색자에게 실제 도움보다 분량을 팔고 있다.
- Google 공식 기준의 "내용 요약" 목적과 다르다.
- 사이트 전체에 같은 문구가 반복돼 복제 느낌을 만든다.

### 2순위: 구식 inline CTA/relatedPosts 스타일 정리

오래된 글에는 inline style로 박힌 CTA가 많다. 이건 콘텐츠 품질 문제이면서 유지보수 문제다.

다만 해결 방향은 "글마다 디자인 다르게"가 아니다.

- 공통 class 또는 렌더러 구조로 정리
- CTA 문구와 위치만 글별로 변주
- 관련글 박스의 역할 라벨을 글별로 변주

### 3순위: 시리즈 H2 템플릿의 복사 사용 차단

최근 30~32번 일주 글은 line 835~839 템플릿을 거의 그대로 따랐다. 문서에는 변주하라고 되어 있지만, 실제 출력은 템플릿 복사에 가깝다.

해결 방향:

- 템플릿은 "포함할 정보 단위"로 바꾼다.
- H2 제목은 매 글 고유 문장으로 재작성하게 한다.
- 같은 시리즈 연속 작성 시 직전 3개 글의 H2와 비교하게 한다.

### 4순위: 결론 문장 패턴 정리

최근 글에도 `결론적으로`, `In short`가 반복된다. 이건 치명적이지는 않지만 AI 패턴 느낌을 만든다.

문서에 "결론/마무리 문장은 반드시 있어야 하는 것이 아니며, 마지막 H2 안에서 자연스럽게 닫아도 된다"는 규칙을 추가하면 좋다.

## 6. 문서 수정 방식 제안

규칙을 길게 더 붙이면 또 안 읽힐 가능성이 높다. 아래 방식이 낫다.

### 방식 A: 위쪽에 "Hard Pattern Ban" 신설

`0-AA` 바로 아래 또는 `0-AC` 앞에 20줄 정도의 강한 금지 목록을 추가한다.

핵심 금지:

- description에 분량 표현 금지
- seoTitle 후위어 반복 금지
- 같은 CTA 문구 연속 사용 금지
- 같은 관련글 박스 라벨 반복 금지
- 시리즈 H2 템플릿 제목 복사 금지
- `결론적으로`, `In short`, `This guide explains` 같은 마무리/도입 패턴 반복 금지

### 방식 B: 기존 섹션에 분산 추가

이 방식은 문서 구조상 자연스럽지만, 실제 작성자가 놓칠 가능성이 있다.

- 5-1 제목: 후위어 제한
- 5-4 메타 설명: 분량 표현 금지
- 6 내부 링크: CTA/관련글 문구 변주
- 8 완료 규칙: 패턴 감사 게이트
- 15 시리즈 템플릿: 복사 금지 강화

추천은 A+B 혼합이다. 위쪽에는 짧은 금지 목록을 두고, 아래 기존 섹션에는 세부 규칙을 넣는 것이 좋다.

## 7. 오프스 분석에 대한 최종 평가

오프스 분석은 방향성이 좋다. 특히 "문서 자체보다 실제 글 수정이 더 시급하다"는 결론은 맞다.

하지만 내가 보기에는 이렇게 정리하는 것이 더 정확하다.

1. `post-codex-command.md`에는 반복 방지 원칙이 이미 충분히 많다.
2. 그래도 오프스가 말한 3개 누락 규칙은 실제로 추가할 가치가 있다.
3. 추가로, 문서에는 "강제 검수 게이트"와 "시리즈 템플릿 충돌 해소"가 더 필요하다.
4. 실제 글은 오래된 누적 글에서 문제가 훨씬 크고, 최근 33~36번도 CTA/결론 패턴은 아직 반복된다.
5. CSS를 글마다 다르게 만들 필요는 없고, 디자인은 공통화하되 문구·위치·역할을 변주해야 한다.
6. 문서 수정 후에는 기존 글 정비 작업을 별도 배치로 진행해야 한다. 그렇지 않으면 새 규칙만 좋아지고 사이트 전체 품질은 그대로 남는다.

## 8. 권장 결론

`post-codex-command.md`는 수정할 필요가 있다. 다만 대규모로 갈아엎을 필요는 없다.

수정 우선순위는 다음이다.

1. `description` 분량 표현 금지 규칙 추가
2. `seoTitle` 후위어 반복 제한 추가
3. CTA/관련글 블록의 문구·위치·역할 변주 규칙 추가
4. 완료 전 패턴 감사 체크리스트 추가
5. `Series H2 Template Rules`를 "복사용 제목"이 아니라 "포함할 정보 단위"로 재정의
6. `Start Here`에 시리즈 글 작성 시 14~15번, 민감 주제 작성 시 18번을 보라는 조건 추가
7. 기존 글 정비 우선순위 규칙 추가

이 정도면 오프스가 지적한 핵심 문제를 문서 차원에서 막을 수 있다. 하지만 문서 수정만으로 끝내면 안 되고, 이후 실제 글 데이터에서 `3,000자`, `완벽 가이드`, inline gradient CTA, 반복 결론 H2를 정리하는 작업이 필요하다.
