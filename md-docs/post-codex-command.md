## 0-AA. Highest Priority Publishing Principle

이 문서에서 앞으로 신규 포스트를 작성할 때 가장 먼저 지켜야 할 원칙은 아래와 같다.

- 이 섹션은 메인 키워드/하위 키워드 배치에 대한 최상위 기준이며, 아래 다른 섹션에 비슷한 표현이 있어도 이 원칙을 우선 적용한다.
- 메인 키워드는 제목/H1, `slug`, `seoTitle`, 첫 문단, 대표 H2, FAQ에 자연스럽게 연결되어야 한다.
- 하위 키워드는 각 H2 주제에 맞게 분산 배치하되, 한 문단에 몰아넣거나 같은 표현을 기계적으로 반복하지 않는다.
- 한글이든 영어든 사용자가 실제 검색창에 입력해 유입될 수 있는 메인 키워드와 하위 키워드를 반드시 넣는다.
- KO/EN 모두 `seoTitle`, `seoTitleEn`, `keywords`, `keywordsEn`, 첫 문단, 대표 H2, FAQ까지 같은 검색 의도를 기준으로 메인 키워드와 하위 키워드를 반영한다.
- 이미지는 절대로 임의로 넣지 않는다.
- 기존 포스트에서 이미 사용한 사진은 절대 다시 사용하지 않는다.
- 아직 사용하지 않았더라도, 이 프로젝트 안에 들어 있는 기존 사진을 임의로 골라 쓰지 않는다.
- 사용 가능한 이미지는 사용자가 직접 전달한 사진만으로 제한한다.
- 사용자가 사진을 주지 않았으면 이미지는 넣지 않는 것을 원칙으로 한다.
- 새 포스트 작성이 끝나면 `md-docs/now-posting.md`의 해당 제목 옆에 반드시 `[완료]`를 붙여 상태를 갱신한다.

- 기존 문서를 억지로 수정하거나 재활용하는 것보다, `실제로 검색되는 키워드`를 기준으로 한 신규 글 발행을 우선한다.
- 앞으로 작성하는 문서는 `검색 가능한 메인 키워드`가 먼저 확정되지 않으면 시작하지 않는다.
- 추상적인 주제, 감으로 좋아 보이는 주제, 검색 수요가 불분명한 주제보다 `사용자가 실제로 검색창에 입력하는 표현`을 우선 채택한다.
- 신규 글은 메인 키워드 1개를 중심으로 잡고, 하위 키워드는 그 메인 키워드를 보완하는 범위에서만 분산 배치한다.
- 제목, slug, 첫 문단, H2, FAQ는 모두 같은 검색 의도를 향해야 하며, 한 글 안에서 여러 검색 의도를 섞지 않는다.
- 앞으로 신규 글 구조는 검색형 글 기준으로 작성하는 것을 원칙으로 한다.
- 기본 구조:
- H1: 키워드 제목
- H2: 기본 개념
- H2: 해석 방법
- H2: 실전 예시
- H2: 자주 묻는 질문

### 0-AA-1. 메인 키워드 / 하위 키워드 배치 규칙

- 메인 키워드는 `H1`, 첫 문단, 최소 1개 `H2`, FAQ 1개에 자연스럽게 들어가야 한다.
- 하위 키워드는 각 H2에 분산 배치하되, 한 문단에 몰아넣지 않는다.
- 같은 키워드를 기계적으로 반복하지 않고, 변형어 / 질문형 / 비교형 / 사례형 표현으로 분산한다.
- `slug`, `seoTitle`, 첫 문단, H2, FAQ가 모두 같은 메인 키워드와 검색 의도를 향하도록 맞춘다.
- 하위 키워드는 메인 키워드를 보조하는 범위에서만 사용하며, 다른 글로 분리해야 할 별도 검색 의도까지 억지로 포함하지 않는다.
- 이 규칙은 한글과 영어 모두 동일하게 적용한다. 영어도 번역문이 아니라 실제 영어권 사용자가 검색할 표현을 기준으로 메인 키워드와 하위 키워드를 넣는다.

### 0-AA-2. 한 글당 검색 의도 1개 원칙

- 한 글은 하나의 메인 검색 의도만 다룬다.
- `보는법`, `뜻`, `궁합`, `특징`, `차이`, `많으면`, `부족하면`, `무료 풀이` 같은 서로 다른 검색 의도는 한 글에 섞지 않는다.
- 여러 검색 의도가 보이면, 메인 의도 1개만 남기고 나머지는 별도 글 후보로 분리한다.
- 제목은 넓고, 본문은 좁은 구조가 되지 않게 한다. 제목에서 약속한 검색 의도를 본문 전체가 그대로 따라가야 한다.

### 0-AA-3. 신규 글 우선순위 원칙

- 앞으로는 기존 글 리라이트보다 `검색형 신규 글 발행`을 우선한다.
- 검색 수요가 명확한 키워드가 있으면, 비슷한 기존 글을 억지로 고치는 것보다 신규 글로 분리 발행하는 쪽을 우선 검토한다.
- 신규 글 우선 패턴:
- `보는법`
- `뜻 / 의미`
- `차이 / 비교`
- `특징`
- `많으면 / 부족하면`
- `무료 / 셀프 / 스스로`
- `예시 / 사례`
- 기존 글 보강은 아래 경우에만 우선한다.
- 이미 같은 검색 의도를 직접 먹고 있는 글이 있고
- slug, seoTitle, 본문 구조를 강화하는 편이 신규 발행보다 효율적일 때

### 0-AA-4. 구조는 유지하되 반복 패턴 금지

- 기본 구조는 유지하되, 모든 글이 같은 문장 길이, 같은 문단 수, 같은 설명 순서로 복제되지 않게 한다.
- 어떤 글은 질문형 도입, 어떤 글은 즉답형 도입, 어떤 글은 오해 교정형 도입처럼 첫 문단 전개를 바꾼다.
- H2 개수는 4~6개 범위 안에서 주제에 맞게 조정한다.
- 실전 예시는 체크리스트형, 사례형, 비교형, 잘못된 해석 vs 올바른 해석형 등으로 변주한다.
- FAQ는 질문 패턴과 답변 길이를 매번 똑같이 맞추지 않는다.
- 결론, CTA, 관련글 연결 문구도 3~4개 이상 패턴으로 분산해 반복 느낌을 줄인다.

### 0-AA-5. AI 티가 나는 반복형 패턴 방지 원칙

- 동일한 H2 순서, 동일한 문단 길이, 동일한 FAQ 패턴, 동일한 CTA 문구를 반복 생산하지 않는다.
- 검색 의도는 유지하되, 전개 방식은 글마다 변주한다.
- 모든 글을 `정의 -> 특징 -> 결론` 같은 단일 공식으로 기계적으로 밀어붙이지 않는다.
- 문장 길이는 짧은 문장, 설명 문장, 예시 문장을 섞어서 리듬을 만든다.
- 실제 검색 사용자가 헷갈리는 지점, 초보자가 막히는 지점, 자주 오해하는 지점을 글마다 다르게 넣는다.
- 사례, 체크리스트, 비교표, 주의 포인트, 해석 실수 예시 같은 장치를 주제에 맞게 선택적으로 사용한다.
- 같은 카테고리 안에서도 각 글의 목적을 분명히 달리한다.
- 예:
- 어떤 글은 입문형
- 어떤 글은 실전형
- 어떤 글은 비교형
- 어떤 글은 오해 교정형
- 검색 봇보다 먼저 사람 독자가 읽기에 덜 복제 같고 더 유용한 문서가 되도록 작성한다.

## 0-AB. Keyword Recommendation Rule

이 문서는 앞으로 `검증된 검색 수요가 확인된 키워드만 추천`하는 것을 강제한다.

### [K1] 검증 안 된 키워드는 추천 금지

[DO]
- 새 글 주제를 추천할 때는 `조회수`, `검색 수요`, `트렌드`, `광고주 수요`, `실제 검색 결과 노출 빈도` 중 최소 1개 이상 근거를 먼저 확인한다.
- 근거를 확인하지 못했으면 `추천하지 않는다`.
- 확실한 근거가 없으면 `아이디어`, `가설`, `보조 주제`로도 제안하지 않는다.

[NOT]
- 감으로 좋아 보이는 키워드를 메인 키워드처럼 추천하지 않는다.
- `조회수 있을 것 같다`, `시즌성이 있을 것 같다` 같은 추측을 검증된 사실처럼 말하지 않는다.
- 검증 없는 키워드를 `다음 글 1순위`로 제안하지 않는다.
- 검증되지 않은 키워드를 brainstorming, 아이디어 제안, 보조 주제 제안 명목으로도 꺼내지 않는다.

### [K2] 메인 키워드 추천 기준

[DO]
- 메인 키워드는 `실제 검색 수요가 강한 것만` 추천한다.
- 가능하면 아래 순서로 확인한다.
1. Search Console 실제 노출/클릭 데이터
2. Google Trends 상대 수요
3. Google 검색 결과 노출 빈도
4. 광고/상업 의도 여부

[NOT]
- Search Console이나 검색 수요 근거 없이 메인 키워드를 정하지 않는다.
- 보조 키워드와 메인 키워드를 섞어서 추천하지 않는다.

### [K3] 답변 방식 규칙

[DO]
- 추천할 때는 반드시 `검증됨` / `미검증`을 구분해서 적는다.
- 사용자가 `조회수 많은 것만` 원하면 미검증 키워드는 아예 제외한다.
- 기존 글과 중복 여부도 같이 확인한다.
- 사용자가 검증 기준을 요구한 경우, 검증되지 않은 항목은 응답 후보군 자체에서 제거한다.

[NOT]
- 검증 여부를 생략하지 않는다.
- 사용자가 엄격한 기준을 요구했는데 아이디어성 키워드를 섞지 않는다.
- 검증되지 않은 내용을 "가능성", "아이디어", "감"으로 포장해서 제시하지 않는다.

### [C1] slug와 주제는 반드시 일치해야 한다

[DO]
- `slug`, `seoTitle`, 본문 첫 문단, H2 구조가 같은 검색 의도를 가리키게 만든다.

[NOT]
- 달력 글 slug에 용신 제목을 넣는 식의 mismatch 금지
- 같은 주제를 다른 slug에서 중복 확장 금지

### [C2] 현재 프로젝트 구조를 기준으로 작업한다

[DO]
- 최종 데이터는 `lib/blog/*-posts.ts` 구조를 기준으로 작성한다.
- 현재 사이트 라우트와 렌더링 방식을 기준으로 판단한다.

[NOT]
- 예전 `.mdx` 워크플로우, 예전 프로젝트 경로, 예전 라우트 규칙을 그대로 가져오지 않는다.

### [C3] KO/EN은 같은 포스트 쌍으로 관리한다

[DO]
- `contentKo`, `contentEn`, `seoTitle`, `seoTitleEn`, `description`, `descriptionEn`, `keywords`, `keywordsEn`, `toc`, `tocEn`, `faq`, `faqEn`을 같이 본다.
- 같은 slug에서 KO/EN 주제가 달라지지 않게 한다.
- 같은 slug에서는 KO/EN 모두 사용자가 검색해서 들어올 수 있는 메인 키워드와 하위 키워드를 각각의 언어 검색 표현에 맞게 반드시 반영한다.

[NOT]
- 한쪽 언어만 다른 검색 의도로 흘러가게 두지 않는다.

### [C4] 카니발리제이션을 먼저 피한다

[DO]
- 새 글 또는 수정 글이 기존 어떤 포스트의 메인 쿼리와 겹치는지 먼저 본다.
- `lucky-element`와 `career-star`처럼 인접한 주제는 의도를 분리한다.

[NOT]
- "좋아 보이는 키워드"를 여러 글에 반복 배치하지 않는다.

### [C5] 이미 해결된 문제를 다시 고치지 않는다

[DO]
- 실제 코드와 `md-docs/seo-residual-checklist.md`를 먼저 확인한다.

## 3. Writing Format Rules

본문 작성 규칙:
- `contentKo`, `contentEn`은 HTML 문자열 형식으로 작성한다.
- 기본 구조는 `<section>`, `<h2 id="...">`, `<h3>`, `<p>`, 필요 시 `<ul><li>`를 사용한다.
- 첫 문단은 추상적 인사말보다 검색 의도를 바로 설명하는 문장으로 시작한다.
- 본문은 정의 -> 해석 원리 -> 실전 적용 -> 주의점 -> 결론 흐름을 기본으로 둔다.

권장 구조:
1. 첫 문단: 검색 의도 즉시 설명
2. 상단 관련글 또는 안내 블록
3. H2 4~6개
4. 중간 CTA 또는 서비스 연결 1회
5. 하단 관련글 2개
6. FAQ 2~3개

금지:
- 모든 H2 아래를 같은 길이, 같은 톤으로 기계적으로 반복
- 의미 없는 미사여구만 길게 늘리기
- 키워드만 나열하는 문장

---

### 3-X. Opening Hook Rules

[DO]
- 첫 2문장 안에서 사용자가 왜 이 키워드를 검색했는지에 대한 불안, 궁금증, 실전 고민을 정확히 짚는다.
- 도입부는 "이 글이 어떤 문제를 해결하는지"를 바로 보여주는 문장으로 시작한다.
- 메인 키워드는 첫 문단 안에 자연스럽게 포함하되, 억지 반복 없이 해결 기대를 만들어야 한다.
- 독자가 "내가 찾던 답이 이 글에 있겠다"라고 느끼도록 문제 인식 -> 해석 방향 순서로 전개한다.

[NOT]
- "오늘은 OO에 대해 알아보겠습니다" 같은 비어 있는 도입 금지
- 추상적인 배경 설명만 길게 늘어놓고 검색 의도를 뒤로 미루는 구성 금지
- 첫 문단에서 메인 키워드만 기계적으로 반복하는 문장 금지

## 5. SEO Rules

### 5-1. 제목

- `seoTitle`은 검색어 의도를 바로 드러내야 한다.
- 클릭만 노린 과장 제목보다, 사용자가 얻을 정보가 명확해야 한다.
- 연도 키워드는 실제 연도성 글에만 사용한다.

### 5-2. 첫 문단

- 메인 키워드 1개와 보조 키워드 1개 정도를 자연스럽게 포함한다.
- 첫 문단에서 "이 글이 무엇을 해결하는지"를 즉시 보여준다.

### 5-3. H2/H3

- H2 중 최소 1개 이상은 메인 검색 의도와 직접 연결되게 잡는다.
- H3는 보조 키워드 또는 실전 해석 단위로 사용한다.

### 5-4. 메타 설명

- 클릭 유도만 하지 말고, 글에서 실제 다루는 범위를 요약한다.
- 제목과 다른 내용을 약속하지 않는다.

### 5-5. 키워드 배열

- `keywords`는 주제 범위를 벗어나지 않게 유지한다.
- 개수보다 주제 정합성이 더 중요하다.

### 5-6. 남아 있는 실제 SEO 우선순위

1. Search Console 기준 실제 카니발리제이션 확인
2. 앵커 텍스트와 `relatedPosts` 정리
3. 얇은 글 후보 실측
4. 허브 페이지 설명력 점검

자세한 내용은:
- `md-docs/seo-residual-checklist.md`

---

### 5-X. Long-Tail Search Intent Rules

[DO]
- 메인 키워드만 반복하지 말고, 사용자가 실제 검색창에 입력할 법한 문장형/질문형 하위 키워드를 발굴해 H2, FAQ, 본문 설명에 분산 반영한다.
- H2 중 일부와 FAQ는 "사람이 실제로 묻는 표현"을 기준으로 작성한다.
- FAQ는 본문 요약이 아니라, 구글 관련 질문(People Also Ask)에서 나올 법한 오해, 비교, 실전 고민, 타이밍 질문을 다룬다.
- 질문형 하위 키워드는 검색 의도에 맞게 배치하되, 모든 H2를 질문형으로 고정하지는 않는다.

[NOT]
- 하위 키워드를 명사 나열 수준으로만 처리하는 방식 금지
- FAQ를 본문 문장을 그대로 바꿔 적는 수준으로 작성 금지
- 모든 소제목을 동일한 질문형 패턴으로 반복하는 방식 금지

### 5-Y. Semantic Keyword Expansion Rules

[DO]
- 메인 키워드를 기계적으로 반복하지 말고, 의미적으로 연결된 연관 개념군을 본문 전체에 자연스럽게 확장한다.
- 하위 키워드는 동의어, 유의어, 관련 상황어, 해석에 필요한 전문 용어, 사용자가 함께 궁금해할 표현까지 포함해 구성한다.
- 예를 들어 `사주 직업운`을 다룰 때는 직업운 자체뿐 아니라 진로 적성, 관성, 조직 적응, 이직 흐름, 천직, 프리랜서 성향 같은 연관 개념을 함께 엮는다.
- KO/EN 모두 각 언어권 사용자가 실제로 쓰는 자연스러운 검색 표현 기준으로 확장한다.

[NOT]
- 메인 키워드만 문단마다 반복하는 방식 금지
- 연관 어휘를 억지로 끼워 넣어 문장이 부자연스러워지는 방식 금지
- 한국어 구조를 영어로 직역한 부자연스러운 키워드 확장 금지

---

## 6. Internal Linking Rules

핵심:
- 링크 포맷보다 더 중요한 것은 링크 문구의 검색 의도 일치다.

원칙:
- 본문 중간 링크와 하단 관련글은 실제 연결 이유가 분명해야 한다.
- 앵커 텍스트는 대상 글의 메인 주제를 흐리지 않게 쓴다.
- `relatedPosts`도 내부 링크 신호의 일부로 본다.

주의:
- 현재 렌더러가 내부 링크를 locale 기준으로 정규화하므로, 소스 문자열 혼재만 보고 문제라고 단정하지 않는다.
- 대신 최종 의도는 locale 일관성과 주제 일관성이다.

앵커 텍스트 예시 원칙:
- `lucky-element` 링크 문구: 용신, 균형, 부족한 기운, 개운
- `career-star` 링크 문구: 관성, 직업운, 조직 적성, 사회적 성공
- `lunar-solar-calendar` 링크 문구: 음력 양력, 절기, 생시, 시간 보정

금지:
- `여기`, `관련글`, `click here`처럼 비어 있는 문구만 쓰기
- 대상 글 메인 의도와 다른 키워드로 반복 호출하기

---

### 6-X. Internal Link Anchor Rules

[DO]
- 내부 링크 앵커 텍스트는 문맥 안에서 클릭 이유가 분명하게 보이도록 작성한다.
- CTA성 링크나 전환 목적 링크는 결과/행동이 드러나는 롱테일 문구를 우선 사용한다.
- 예: `내 사주에 부족한 오행 균형 잡는 방법 알아보기`, `직업운 해석 전에 관성을 먼저 확인하는 법 보기`
- 설명형 본문 링크는 짧은 앵커도 허용하되, 문맥만 읽어도 왜 이동하는지 이해되게 만든다.

[NOT]
- 모든 링크를 1~2단어짜리 기계적 앵커로 처리하는 방식 금지
- 반대로 모든 링크를 과도하게 긴 문장형 앵커로 만들어 가독성을 해치는 방식 금지
- `여기`, `자세히 보기`, `click here` 같은 빈 앵커 금지

---

## 7. FAQ / Structured Data Rules

- FAQ는 2~3개 정도를 기본으로 본다.
- 질문은 실제 검색자가 던질 법한 형태로 쓴다.
- 답변은 본문 반복이 아니라 예외, 오해, 실전 적용을 보완해야 한다.
- 현재 블로그 포스트 페이지는 FAQ JSON-LD를 출력하고 있으므로, 데이터 품질이 더 중요하다.

주의:
- FAQ 리치결과 노출은 자동 보장으로 보면 안 된다.
- FAQ가 있다고 CTR이 무조건 오른다고 가정하지 않는다.

## 8. Completion Rules

포스트 작성 또는 수정 후 아래를 확인한다.

1. slug와 실제 주제가 일치하는지
2. KO/EN 제목과 본문 의도가 같은지
3. `toc`와 `<h2 id>`가 일치하는지
4. `description`이 실제 본문을 요약하는지
5. `faq`가 추가 정보를 주는지
6. `relatedPosts`와 앵커 텍스트가 주제와 맞는지
7. 기존 포스트와 키워드 충돌이 없는지
8. 필요하면 `updatedAt` 반영 여부를 판단했는지
9. 운영 문서 업데이트가 필요한지 확인했는지

업데이트 대상 문서:
- `md-docs/now-posting.md`
- `md-docs/seo-residual-checklist.md`
- 필요 시 `md-docs/haha-imsi`

## 10. Image Rules

- 대표 이미지와 본문 설명 이미지는 검색 유입과 체류시간에 도움될 수 있다.
- 이미지 추가 자체는 애드센스 승인 후에도 문제 아님.
- 다만 의미 없는 장식 이미지 남발은 피한다.
- 게시용 이미지는 최종적으로 `.webp`만 남긴다.
- 원본 파일이 `png`, `jpg`, `jpeg` 등이라면 `.webp` 변환 후 게시 경로 반영이 끝난 다음 `.webp` 이외 원본 파일은 모두 삭제한다.
- 본문, 문서, 코드에는 `.webp` 이외 확장자를 최종 경로로 남기지 않는다.

원칙:
- 글 주제와 직접 연결된 이미지만 넣는다.
- 용량 최적화가 우선이다.
- 파일명과 `alt`는 검색 의도와 맞게 쓴다.
- 핵심 글부터 순차적으로 추가한다.

## 11. What Not To Overreact To

- "애드센스 승인 후 수정하면 안 된다"는 식의 속설
- 이미 해결된 문제를 계속 1순위로 보는 것
- FAQ 리치결과를 자동 기대하는 것
- 샌드박스 기간 숫자를 확정 사실처럼 다루는 것

현실적인 원칙:
- 대수술은 피하되, 검색 의도 정렬과 품질 개선은 바로 한다.

## 12. Default Work Order

포스트 작업 기본 순서:

1. 기존 유사 포스트 확인
2. slug와 검색 의도 확정
3. `seoTitle`, `description`, `keywords` 초안 작성
4. KO/EN 본문 작성 또는 수정
5. `toc`, `faq`, `relatedPosts` 정리
6. 내부 링크 문구 점검
7. 카니발리제이션 재확인
8. 운영 문서 반영 필요 여부 판단

## 13. One-Line Principle

이 프로젝트에서 가장 중요한 것은 "많이 쓰는 것"보다 "각 포스트가 어떤 검색 의도를 담당하는지 명확하게 유지하는 것"이다.
